 Product Requirements Document: LLM Model & Session Management

 Document Version: 1.0
 Date: 2026-05-27
 Project: Gemini Clone

 ────────────────────────────────────────────────────────────────────────────────

 ### 1. Introduction

 This document outlines the features and functionality implemented to enhance the LLM-powered chat application. The
 primary goal was to introduce greater configurability for Language Model (LLM) providers and to provide a more
 structured way to manage and visualize chat sessions, differentiating between system-generated Pi agent sessions and
 user-initiated chat sessions.

 ### 2. Problem Statement

 Previously, the application relied on a single, environment-variable-configured LLM model, lacked a user-friendly
 mechanism to add, remove, or switch between multiple models, and did not support custom metadata or logging for model
 interactions. Furthermore, all session logs, regardless of their origin (Pi agent or direct chat interactions), were
 commingled, making it difficult to distinguish and manage different types of conversations.

 ### 3. Solution Overview

 The solution introduces a flexible, JSON-based configuration system for LLM models, allowing users to define multiple
 models with custom metadata and logging paths. It provides a dedicated user interface in the frontend for managing
 these models. Additionally, the session management has been overhauled to clearly separate Pi agent sessions from
 custom chat sessions, presenting them in distinct areas of the navigation and providing dedicated viewing/interaction
 flows.

 ### 4. Scope

 This PRD covers the following key areas:

 - LLM Model Configuration: Defining a structure for LLM models and managing them.
 - Model Management UI: Frontend interface for interacting with model configurations.
 - Custom Model Data: Support for arbitrary JSON metadata and dynamic logging paths.
 - Session Management: Unique session IDs, logging, and separated display of session types.

 ### 5. Detailed Features

 #### 5.1. LLM Model Management

 - Configurable models.json:
     - A models.json file is introduced at the project root (/workspace/backend/models.json) to store LLM model
       configurations.
     - This file adheres to a simplified "LLM library friendly format" with model_id, model_name, and active status.
     - It is included in .gitignore to prevent accidental sharing of sensitive information (e.g., API keys if they were
       to be included).
     - The file is initialized with default free Gemini models (gemini-1.5-flash, gemini-1.5-pro, gemini-1.5-flash-8b)
       upon first access.
 - Backend API Endpoints (/api/models):
     - GET /api/models: Retrieves a list of all configured models and the currently active model.
     - POST /api/models: Adds a new model to the configuration. Includes model_id and model_name, with default log_path
        and custom fields.
     - PUT /api/models/{model_id:path}: Updates an existing model's model_name, log_path, and custom metadata. The
       {model_id:path} allows for model IDs containing slashes (e.g., gemini/gemini-2.5-flash).
     - DELETE /api/models/{model_id:path}: Removes a model from the configuration. If the active model is deleted, the
       first available model becomes active.
     - POST /api/models/select/{model_id:path}: Sets a specified model as the active model for chat interactions.
 - Frontend UI (Models Tab):
     - A "Models" navigation tab (with a Settings icon) provides access to the model management interface.
     - Displays a list of all configured models, highlighting the active one.
     - Add Model Form: Allows users to input a model_id and model_name to add new models.
     - Edit Model Inline View: Clicking "Edit" on a model reveals an inline form to:
         - Modify the model_name.
         - Configure log_path (see next section).
         - Edit custom JSON data (see next section).
     - Select/Delete Actions: Buttons to set a model as active or delete it from the configuration.

 #### 5.2. Custom Model Data & Logging

 - log_path Field:
     - Each model configuration can specify an optional log_path (e.g., backend/logs_{session_id}.jsonl).
     - The backend dynamically resolves this path relative to the workspace root.
     - Supports the {session_id} placeholder, which is replaced by the current chat session's UUID during logging.
 - custom JSON Field:
     - An optional custom field allows users to store arbitrary key-value JSON pairs (e.g., {"is_paid": true,
       "provider": "Google"}) within each model's configuration.
     - The frontend's edit view provides a textarea for JSON input, with basic validation.
 - Chat Logging (backend/main.py):
     - The /api/chat endpoint now checks the active model's log_path.
     - If configured, it logs each prompt-response pair to the specified .jsonl file.
     - Each log entry includes timestamp, model_id, session_id, prompt, and response.
     - All *.jsonl files are added to .gitignore.
 - Frontend Rendering of Custom Data:
     - In the "Models" tab, custom data is rendered as a vertical list of formatted key-value pairs (e.g., [key]
       value).

 #### 5.3. Session Management & Separation

 - UUID-based Chat Sessions:
     - Each new chat initiated by the user is assigned a unique UUID (sessionId) in the frontend.
     - This sessionId is passed to the backend with each chat request.
 - + New Chat Button:
     - A "New" button is added next to the "Chat" navigation link in the sidebar.
     - Clicking it clears the current chat messages and generates a new sessionId, effectively starting a new
       conversation.
 - Separated Session Displays:
     - Pi Agent Sessions: Sessions generated by the Pi agent are displayed under the dedicated "Sessions" tab in the
       main content area.
     - Custom Chat Sessions: Sessions logged via the custom log_path are displayed as a historical list directly in the
       sidebar, nested under the "Chat" navigation link.
 - Unified Session Fetching (backend/main.py):
     - The GET /api/sessions endpoint now aggregates sessions from two sources:
         1. The standard Pi agent session directory (~/.pi/agent/sessions/--workspace--/).
         2. Custom chat logs from log_path configurations (using glob patterns to find files matching {session_id} in
            their path).
     - Each session object returned now includes a source field ("pi" or "chat") for frontend differentiation.
     - For chat sessions, a title (preview of the first prompt) is generated for display in the sidebar.
 - Loading Specific Chat Sessions:
     - Clicking a custom chat session from the sidebar list loads its entire history into the main chat interface,
       allowing the user to view and potentially continue the conversation. The UI correctly reconstructs user and
       assistant messages from the jsonl logs.

 ### 6. Technical Details

 - Backend Stack: FastAPI, llm library (for model interaction), json for configuration, glob and pathlib for file
   system operations, datetime for timestamps.
 - Frontend Stack: React (with useState, useEffect, useRef hooks), Axios for API interaction, Tailwind CSS for styling,
   Lucide icons.
 - Data Persistence: models.json for model configurations, .jsonl files for chat session logs.
 - API Path Handling: model_id in API routes uses :path converter to correctly handle slashes.
 - Error Handling: Basic error handling and user alerts for model operations and API communication.

 ### 7. Future Considerations

 - Streaming Responses: Implementing real-time streaming for LLM responses to improve user experience.
 - More Advanced Model Parameters: Allowing configuration of additional model parameters (temperature, max tokens,
   etc.) directly in models.json and the UI.
 - Session Management UI/UX: Enhanced filtering, searching, and archiving capabilities for sessions.
 - Multi-turn Chat Persistence: Storing full chat history in models.json for more robust continuation across backend
   restarts.
 - Tool Usage Logging: Extending jsonl logging to capture tool calls and their results.
